Introduction

Welcome to Lesson 2 of CP363: Database I. In this lesson, we will explore Entity-Relationship (ER) modeling, a fundamental technique for conceptual database design. ER modeling provides a way to visualize the structure of a database before implementing it, helping to ensure that the database accurately represents the real-world system it's modeling.

Database design is a critical phase in database development. A well-designed database is easier to implement, maintain, and modify as requirements change. It also provides better performance and data integrity. Entity-Relationship modeling is the first step in this design process, allowing us to create a high-level conceptual model of the data and relationships that will be stored in the database.

The ER model was first proposed by Peter Chen in 1976 and has since become one of the most widely used approaches for conceptual database design. It provides a set of concepts and notations that allow database designers to capture the semantics of real-world data in a graphical representation called an Entity-Relationship Diagram (ERD).

In this lesson, we will learn about the database design process, the core concepts of the ER model (entities, attributes, relationships), different notations for ER diagrams, and practical techniques for creating effective ER models. By the end of this lesson, you will be able to create ER diagrams for various scenarios and understand how these diagrams serve as a blueprint for database implementation.

Let's begin our exploration of Entity-Relationship modeling!

Part 1: Database Design Process

Before diving into the specifics of ER modeling, it's important to understand where it fits within the overall database design process. Database design typically follows a structured approach consisting of several phases:

Requirements Collection and Analysis

The database design process begins with understanding what the database needs to accomplish. This involves:

  • Interviewing stakeholders: Meeting with users, managers, and other stakeholders to understand their needs and expectations.
  • Analyzing existing systems: Examining current data storage methods, forms, reports, and processes.
  • Documenting requirements: Creating detailed documentation of data requirements, including what data needs to be stored, how it will be used, and any constraints or business rules.
  • Defining scope: Determining the boundaries of the database system—what it will and won't include.

The output of this phase is a set of requirements specifications that serve as the foundation for the subsequent design phases.

Conceptual Database Design

This is where Entity-Relationship modeling comes into play. The conceptual design phase involves:

  • Identifying entities: Determining the main objects or concepts about which data will be stored.
  • Defining attributes: Specifying the properties or characteristics of each entity.
  • Establishing relationships: Determining how entities are related to each other.
  • Creating ER diagrams: Representing the conceptual model graphically using ER notation.

The conceptual design is independent of any specific database management system (DBMS) or implementation details. It focuses on capturing the semantics of the real-world system being modeled.

The primary output of this phase is an Entity-Relationship Diagram (ERD) that represents the conceptual schema of the database.

Logical Database Design

Once the conceptual design is complete, it is translated into a logical model that can be implemented in a specific type of database system. For relational databases, this involves:

  • Mapping entities to tables: Converting each entity into a table (relation).
  • Mapping attributes to columns: Converting each attribute into a column in the appropriate table.
  • Mapping relationships: Implementing relationships through foreign keys or relationship tables.
  • Normalization: Applying normalization techniques to eliminate redundancy and ensure data integrity.
  • Defining constraints: Specifying primary keys, foreign keys, unique constraints, and other integrity constraints.

The logical design is specific to a type of database model (e.g., relational, document, graph) but is still independent of any specific DBMS product.

The output of this phase is a logical schema, typically represented as a set of table definitions with columns, data types, and constraints.

Physical Database Design

The final phase of database design involves optimizing the logical design for a specific DBMS and hardware environment. This includes:

  • Selecting storage structures: Determining how data will be physically stored.
  • Creating indexes: Defining indexes to improve query performance.
  • Implementing security: Setting up user accounts, roles, and permissions.
  • Tuning for performance: Optimizing the database for specific workloads and hardware configurations.
  • Planning for growth: Considering future data volume and usage patterns.

The physical design is specific to a particular DBMS product (e.g., MySQL, Oracle, SQL Server) and hardware environment.

The output of this phase is a physical schema that can be implemented directly in the target DBMS, including SQL statements for creating tables, indexes, views, and other database objects.

In this lesson, we will focus primarily on the conceptual design phase, specifically on Entity-Relationship modeling. The ER model we create will serve as the foundation for the logical and physical design phases that follow.

Part 2: ER Model Concepts

The Entity-Relationship model consists of several key concepts that allow us to represent the structure and semantics of data. Let's explore each of these concepts in detail:

Entities and Entity Sets

Entity: An entity is a real-world object, concept, or event about which we want to store data. Examples include a person, a place, a product, an order, or an event.

Entity Set: An entity set is a collection of entities of the same type. For example, all students in a university form the "Student" entity set, and all courses form the "Course" entity set.

In ER diagrams, entity sets are typically represented as rectangles labeled with the name of the entity set.

Types of Entities:

  • Strong Entity: An entity that can exist independently of other entities. It has its own primary key that uniquely identifies each entity instance. Examples include Student, Employee, and Department.
  • Weak Entity: An entity that cannot exist independently and relies on another entity (its owner or identifying entity) for its identification. A weak entity does not have a complete primary key of its own. Examples include Dependent (of an Employee) or Room (in a Building).

Viewing Types of Entities

Attributes and Their Types

Attribute: An attribute is a property or characteristic of an entity. For example, a Student entity might have attributes such as student_id, name, date_of_birth, and email.

In ER diagrams, attributes are typically represented as ovals connected to their entity rectangle.

Types of Attributes:

  • Simple Attribute: An attribute that cannot be divided further. Examples include age, gender, or salary.
  • Composite Attribute: An attribute that can be divided into smaller subparts. For example, address can be divided into street, city, state, and zip code.
  • Single-valued Attribute: An attribute that holds only one value for each entity instance. Examples include social security number or date of birth.
  • Multi-valued Attribute: An attribute that can hold multiple values for each entity instance. Examples include phone numbers (a person may have multiple phone numbers) or skills (an employee may have multiple skills).
  • Derived Attribute: An attribute whose value can be calculated from other attributes. Examples include age (derived from date of birth and current date) or total_price (derived from quantity and unit_price).
  • Stored Attribute: An attribute whose value is explicitly stored in the database, as opposed to being derived.
  • Complex Attribute: An attribute that combines multiple types, such as a multi-valued composite attribute.

In ER diagrams, different types of attributes are represented with different notations:

  • Simple attributes: Regular ovals
  • Composite attributes: Ovals connected to other ovals (subattributes)

    • Viewing Composite Attributes

  • Multi-valued attributes: Double ovals
  • Derived attributes: Dashed ovals

Viewing Types of Attributes

Keys

Keys are special attributes or combinations of attributes that uniquely identify entity instances:

  • Candidate Key: An attribute (or combination of attributes) that can uniquely identify each entity instance in an entity set. An entity set may have multiple candidate keys.
  • Primary Key: The candidate key chosen to be the main identifier for an entity set. In ER diagrams, primary key attributes are often underlined.
  • Composite Key: A key that consists of multiple attributes.
  • Partial Key: An attribute (or combination of attributes) that can uniquely identify weak entity instances that are related to the same owner entity. A partial key combined with the primary key of the owner entity forms the primary key of the weak entity.

For example, in a Student entity set, student_id might be the primary key, while email could be another candidate key (assuming each student has a unique email address).

Relationships and Relationship Sets

Relationship: A relationship is an association between two or more entities. For example, a Student "enrolls in" a Course, or an Employee "works in" a Department.

Relationship Set: A relationship set is a collection of relationships of the same type. For example, all enrollment relationships between students and courses form the "Enrolls_In" relationship set.

In ER diagrams, relationship sets are typically represented as diamonds labeled with the name of the relationship.

Degree of a Relationship:

  • Unary Relationship (Recursive): A relationship between entities of the same entity set. For example, an Employee "manages" another Employee.

    • Viewing Recursive Relationship

  • Binary Relationship: A relationship between entities of two different entity sets. For example, an Employee "works for" a Department.

    • Viewing Binary Relationship

  • Ternary Relationship: A relationship between entities of three different entity sets. For example, a Project "order" a Part from a Supplier.

    • Viewing Ternary Relationship

  • n-ary Relationship: A relationship between entities of n different entity sets.

Attributes of Relationships: Relationships can also have attributes. For example, the "Enrolls_In" relationship between Student and Course might have attributes like enrollment_date and grade.

Cardinality and Participation Constraints

Constraints specify rules that the data in the database must satisfy. Two important types of constraints in ER modeling are cardinality and participation constraints:

Cardinality Constraints: Cardinality specifies the number of entities to which another entity can be related via a relationship set. Common cardinality ratios include:

  • One-to-One (1:1): Each entity in the first entity set is related to at most one entity in the second entity set, and vice versa. For example, A manager can manage only one department. Each department can have only one manager .

    • Viewing one-to-one

  • One-to-Many (1:N): Each entity in the first entity set can be related to many entities in the second entity set, but each entity in the second entity set is related to at most one entity in the first entity set. For example, A department can have several employees. Each employee may work in only one department.

    • Viewing one-to-many

  • Many-to-Many (M:N): Each entity in the first entity set can be related to many entities in the second entity set, and vice versa. For example, a supplier can supply parts to several projects. A project can receive parts from several suppliers.

    • Viewing Many-to-Many

Participation Constraints: Participation constraints specify whether all entities in an entity set must participate in a relationship set. There are two types:

  • Total Participation (Mandatory): Every entity in the entity set must participate in at least one relationship in the relationship set. In ER diagrams, this is often represented by a double line connecting the entity set to the relationship. For example, every Employee must work in a Department.
  • Partial Participation (Optional): Some entities in the entity set may not participate in any relationship in the relationship set. In ER diagrams, this is represented by a single line. For example, Not every employee manages a department. Employee participation in manages relationship is partial

    • Viewing Partial Participation

Together, cardinality and participation constraints provide a powerful way to express the rules and semantics of the data being modeled.

Weak Entities

As mentioned earlier, a weak entity is an entity that cannot exist independently and relies on another entity (its owner) for its identification. Let's explore this concept in more detail:

Characteristics of Weak Entities:

  • A weak entity does not have a complete primary key of its own.
  • It depends on a strong entity (owner) for its existence.
  • It has a partial key that can uniquely identify weak entity instances related to the same owner entity.
  • The primary key of a weak entity is the combination of the primary key of its owner entity and its own partial key.
  • A weak entity has total participation in its identifying relationship with the owner entity.

Identifying Relationship: The relationship between a weak entity and its owner entity is called an identifying relationship. In ER diagrams, weak entities are often represented as double-bordered rectangles, and identifying relationships as double-bordered diamonds.

Example: Consider a company database where each Employee can have multiple Dependents (e.g., spouse, children). The Dependent entity is a weak entity because:

  • A Dependent cannot exist in the database without being associated with an Employee.
  • The name of a Dependent alone is not enough to uniquely identify them (multiple employees might have dependents with the same name).
  • The primary key of Dependent would be the combination of employee_id (from the Employee entity) and dependent_name (the partial key of Dependent).

In this example, "Has_Dependent" would be the identifying relationship between Employee and Dependent.

Part 3: ER Diagram Notation

Entity-Relationship diagrams (ERDs) provide a visual representation of the ER model. Several notations have been developed for drawing ERDs, each with its own symbols and conventions. Let's explore the most common notations:

Chen Notation

The original ER notation was proposed by Peter Chen in 1976 and is still widely used in academic settings:

  • Entities: Represented as rectangles labeled with the entity name.
  • Attributes: Represented as ovals connected to their entity or relationship by a line. Primary key attributes are underlined.
  • Relationships: Represented as diamonds labeled with the relationship name.
  • Cardinality: Indicated by placing "1" or "N" (or "M") near the lines connecting entities to relationships.
  • Weak Entities: Represented as double-bordered rectangles.
  • Identifying Relationships: Represented as double-bordered diamonds.
  • Total Participation: Represented by a double line connecting an entity to a relationship.
  • Partial Participation: Represented by a single line.

A diagram showing Chen notation with entities (rectangles), attributes (ovals), relationships (diamonds), and various constraints.

Crow's Foot Notation

Crow's Foot notation is widely used in industry and is supported by many database design tools:

  • Entities: Represented as rectangles containing the entity name and often listing the attributes inside the rectangle.
  • Relationships: Represented as lines connecting entities, with the relationship name written on the line.
  • Cardinality: Indicated by symbols at the ends of relationship lines:
    • One: A vertical line |
    • Many: A crow's foot (three lines resembling a bird's foot) ⋉
    • Zero or One: A circle and a vertical line ○|
    • Zero or Many: A circle and a crow's foot ○⋉
    • One or Many: A vertical line and a crow's foot |⋉
  • Primary Keys: Indicated by placing "PK" next to the attribute or by underlining the attribute.
  • Foreign Keys: Indicated by placing "FK" next to the attribute.

Crow's Foot notation does not explicitly represent weak entities or identifying relationships, but these concepts can still be modeled using the available notation.

A diagram showing Crow's Foot notation with entities (rectangles) connected by lines with different symbols at the ends to indicate cardinality.

UML Class Diagram Notation

Unified Modeling Language (UML) class diagrams can also be used to represent ER models:

  • Entities: Represented as classes (rectangles divided into three sections: class name, attributes, and methods). For ER modeling, the methods section is typically empty or omitted.
  • Attributes: Listed in the middle section of the class rectangle, often with their data types.
  • Relationships: Represented as lines connecting classes, with different line styles for different types of relationships:
    • Association: Solid line
    • Aggregation: Solid line with an empty diamond at one end
    • Composition: Solid line with a filled diamond at one end
    • Generalization/Specialization: Solid line with an empty triangle pointing to the superclass
  • Cardinality: Indicated by multiplicity expressions (e.g., "1", "0..1", "1..*", "*") placed near the ends of relationship lines.
  • Primary Keys: Often indicated by stereotypes like «PK» or by underlining the attribute.

UML class diagrams are more general-purpose than traditional ER notations and can represent additional object-oriented concepts that are not part of the basic ER model.

A diagram showing UML class notation with classes (rectangles divided into sections) connected by different types of lines to indicate relationships.

Comparison of Notations

Each notation has its strengths and weaknesses:

  • Chen Notation:
    • Strengths: Explicitly represents all ER concepts (entities, attributes, relationships, constraints); visually distinguishes between different types of elements.
    • Weaknesses: Can become cluttered with many attributes; less commonly used in industry tools.
  • Crow's Foot Notation:
    • Strengths: Compact representation; clearly shows cardinality constraints; widely supported by database design tools.
    • Weaknesses: Does not explicitly represent some ER concepts (e.g., weak entities, identifying relationships).
  • UML Class Diagram Notation:
    • Strengths: Part of a broader modeling language; can represent both data and behavior; widely used in software engineering.
    • Weaknesses: Not specifically designed for database modeling; some ER concepts require additional stereotypes or annotations.

The choice of notation often depends on the specific context, available tools, and personal or organizational preferences. In this course, we will primarily use Chen notation for its educational value in clearly representing all ER concepts, but we will also introduce Crow's Foot notation due to its widespread use in industry.

Part 4: Practical ER Modeling

Now that we understand the concepts and notations of ER modeling, let's explore the practical aspects of creating ER diagrams for real-world scenarios:

Identifying Entities, Attributes, and Relationships

Creating an ER model begins with identifying the key components of the system being modeled:

Identifying Entities:

  • Look for nouns in the requirements that represent distinct objects or concepts.
  • Consider what information needs to be stored and tracked over time.
  • Determine if each potential entity has multiple instances and unique identifiers.
  • Examples: Customer, Product, Order, Employee, Department.

Identifying Attributes:

  • Look for properties or characteristics of the identified entities.
  • Determine what information needs to be stored about each entity.
  • Identify which attributes can serve as keys.
  • Examples: For a Customer entity, attributes might include customer_id, name, address, phone, and email.

Identifying Relationships:

  • Look for verbs or associations between entities in the requirements.
  • Determine how entities interact or are related to each other.
  • Identify the cardinality and participation constraints for each relationship.
  • Examples: Customer "places" Order, Employee "works in" Department, Student "enrolls in" Course.

Example Process:

Consider the following requirement: "A university needs a database to track students, courses, and enrollments. Each student has a unique ID, name, date of birth, and email. Each course has a unique code, title, credit hours, and is offered by a department. Students can enroll in multiple courses, and each enrollment has a semester and a grade."

From this requirement, we can identify:

  • Entities: Student, Course, Department
  • Attributes:
    • Student: student_id (key), name, date_of_birth, email
    • Course: course_code (key), title, credit_hours
    • Department: dept_id (key), name
  • Relationships:
    • Student "enrolls in" Course (M:N relationship with attributes semester and grade)
    • Department "offers" Course (1:N relationship)

Common Design Patterns

Certain patterns appear frequently in ER modeling. Recognizing these patterns can help streamline the design process:

Master-Detail Pattern:

  • A one-to-many relationship between a "master" entity and multiple "detail" entities.
  • Example: Order (master) and Order_Item (detail). An order can have multiple items, but each item belongs to exactly one order.
  • Implementation: The detail entity typically includes a foreign key referencing the master entity.

Associative Entity Pattern:

  • Used to represent a many-to-many relationship with additional attributes.
  • Example: Student and Course have a many-to-many relationship, with Enrollment as an associative entity that includes attributes like semester and grade.
  • Implementation: The associative entity typically has a composite primary key consisting of the primary keys of the two related entities.

Subtype/Supertype Pattern:

  • Used to represent inheritance or specialization/generalization relationships.
  • Example: Person (supertype) with subtypes Employee and Customer. Each employee and customer is also a person and inherits person attributes.
  • Implementation: Can be implemented in various ways, including single table inheritance, table per class, or table per concrete class.

Recursive Relationship Pattern:

  • A relationship between entities of the same type.
  • Example: Employee "manages" Employee. One employee can manage multiple employees, but each employee has at most one manager.
  • Implementation: Typically implemented using a self-referencing foreign key.

Weak Entity Pattern:

  • Used when an entity's existence depends on another entity.
  • Example: Employee and Dependent. A dependent cannot exist without an associated employee.
  • Implementation: The weak entity's primary key includes the primary key of the owner entity.

Common Mistakes in ER Modeling

Even experienced designers can make mistakes when creating ER models. Being aware of common pitfalls can help avoid them:

Confusing Entities and Attributes:

  • Mistake: Modeling something as an attribute when it should be an entity, or vice versa.
  • Example: Modeling "address" as a simple attribute of Customer when it should be a separate entity with its own attributes (street, city, state, zip).
  • Solution: Consider whether the item has its own properties or if multiple instances of it need to be tracked. If so, it's likely an entity rather than an attribute.

Incorrect Cardinality:

  • Mistake: Misidentifying the cardinality of a relationship.
  • Example: Modeling the relationship between Student and Course as one-to-many when it should be many-to-many.
  • Solution: Carefully analyze the real-world relationship and consider multiple scenarios to determine the correct cardinality.

Missing or Redundant Relationships:

  • Mistake: Omitting necessary relationships or including redundant ones.
  • Example: Creating direct relationships A→B, B→C, and A→C when the A→C relationship is redundant and can be derived from the other two.
  • Solution: Ensure that all required relationships are included, but avoid creating relationships that can be derived from others.

Inappropriate Use of Weak Entities:

  • Mistake: Modeling an entity as weak when it should be strong, or vice versa.
  • Example: Modeling Department as a weak entity of Company when it should be a strong entity with its own primary key.
  • Solution: Only use weak entities when an entity's existence truly depends on another entity and it cannot be uniquely identified without its owner.

Overcomplicating the Model:

  • Mistake: Creating an unnecessarily complex model with too many entities and relationships.
  • Example: Creating separate entities for every minor concept in the system.
  • Solution: Strive for simplicity while ensuring the model accurately represents the requirements. Focus on the essential entities and relationships.

Ignoring Business Rules:

  • Mistake: Failing to incorporate important business rules and constraints into the model.
  • Example: Not capturing the rule that an employee can work in only one department at a time.
  • Solution: Carefully analyze the requirements to identify all relevant business rules and ensure they are reflected in the model through appropriate cardinality and participation constraints.

Case Study: University Database

Let's apply what we've learned to design an ER model for a university database:

Requirements:

  • The university has multiple departments, each with a unique ID, name, location, and a chairperson who is a professor.
  • Each department offers multiple courses. A course has a unique code, title, description, credit hours, and level.
  • Professors work in departments. Each professor has an ID, name, rank, office location, and phone number. A professor can work in only one department.
  • Students are enrolled in the university. Each student has an ID, name, date of birth, address, and major department.
  • Students can enroll in multiple courses, and each course can have multiple students. For each enrollment, the semester and grade are recorded.
  • Each course can have multiple sections in a semester. Each section has a unique section number, a classroom, meeting days and times, and a professor who teaches it.
  • Students enroll in specific sections of courses, not just the courses themselves.

ER Model Design:

1. Identify Entities:

  • Department
  • Course
  • Professor
  • Student
  • Section
  • Enrollment (associative entity between Student and Section)

2. Identify Attributes:

  • Department: dept_id (key), name, location
  • Course: course_code (key), title, description, credit_hours, level
  • Professor: prof_id (key), name, rank, office_location, phone
  • Student: student_id (key), name, date_of_birth, address
  • Section: section_id (key), section_number, classroom, meeting_days, meeting_times, semester
  • Enrollment: grade

3. Identify Relationships:

  • Department "offers" Course (1:N)
  • Department "employs" Professor (1:N)
  • Department "has as chairperson" Professor (1:1)
  • Student "has major" Department (N:1)
  • Course "has" Section (1:N)
  • Professor "teaches" Section (1:N)
  • Student "enrolls in" Section (M:N) with attribute grade

A comprehensive ER diagram showing the university database with entities, attributes, relationships, and constraints as described above.

Discussion:

This ER model captures the key entities and relationships in the university database. Some notable aspects of the design include:

  • The chairperson relationship between Department and Professor is a 1:1 relationship, but it's also a subset of the "employs" relationship (a chairperson must be a professor in the department).
  • The Section entity represents specific offerings of a course in a particular semester, allowing the same course to be offered multiple times with different professors, schedules, and classrooms.
  • The Enrollment associative entity captures the many-to-many relationship between Student and Section, with grade as an attribute of the relationship.
  • The model includes both structural relationships (e.g., Department "offers" Course) and participatory relationships (e.g., Professor "teaches" Section).

This case study demonstrates how ER modeling can be used to design a database that accurately represents a complex real-world system with multiple interrelated entities and relationships.

Glossary of Terms

  • Attribute: A property or characteristic of an entity.
  • Cardinality: The number of entities to which another entity can be related via a relationship set.
  • Composite Attribute: An attribute that can be divided into smaller subparts.
  • Conceptual Design: The phase of database design that focuses on creating a high-level model of the data and relationships, independent of implementation details.
  • Derived Attribute: An attribute whose value can be calculated from other attributes.
  • Entity: A real-world object, concept, or event about which we want to store data.
  • Entity Set: A collection of entities of the same type.
  • Entity-Relationship Diagram (ERD): A graphical representation of an ER model.
  • Entity-Relationship Model: A conceptual data model that views the real world as entities and relationships.
  • Identifying Relationship: The relationship between a weak entity and its owner entity.
  • Key: An attribute or combination of attributes that uniquely identifies an entity within an entity set.
  • Logical Design: The phase of database design that focuses on mapping the conceptual model to a specific data model (e.g., relational, document, graph).
  • Multi-valued Attribute: An attribute that can hold multiple values for each entity instance.
  • Participation Constraint: A constraint that specifies whether all entities in an entity set must participate in a relationship set.
  • Physical Design: The phase of database design that focuses on optimizing the logical design for a specific DBMS and hardware environment.
  • Primary Key: The candidate key chosen to be the main identifier for an entity set.
  • Relationship: An association between two or more entities.
  • Relationship Set: A collection of relationships of the same type.
  • Simple Attribute: An attribute that cannot be divided further.
  • Strong Entity: An entity that can exist independently of other entities.
  • Weak Entity: An entity that cannot exist independently and relies on another entity for its identification.
